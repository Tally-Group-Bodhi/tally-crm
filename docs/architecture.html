  • Overview


The purpose of this document is to detail the Tally CIS solution blueprint for the replacement of Salesforce by CS Energy. The Tally CIS solution includes the systems and functional processes to enable customer case management, customer onboarding and other related functions currently performed in Salesforce, for CS Energy’s C&I (Commercial and Industrial) customer base. This also includes integrations for the Solution which will be required for the implementation.   


The blueprinting exercise defines the scope and design of the service solution, this includes:

  • Background
  • Non-functional requirements
  • Functional requirements
  • Business rules
  • Customer Correspondence
  • Reporting
  • APIs
  • Training
  • Transition to go live considerations

The solution blueprint will:

  • Ensure a clear understanding of the solution requirements
  • Capture all relevant information for the implementation


The following items are out of scope of this document:

  • Requirements to support customers outside of the QLD jurisdiction
  • Requirements to service other utilities e.g., Gas
  • Requirements to service off market embedded networks
  • Requirements to service non-Australian markets
    • Executive Summary

CS Energy is a C&I retailer based in Queensland and owned by the QLD Government. CS Energy currently retail electricity in just the QLD jurisdiction.  


CS Energy’s key functions include:

  • Marketing and selling retail energy products
  • Customer management
  • Processing market transfers, new connections and other requested service orders for customers 
  • Providing customers with timely and accurate billing (with the Support of Tally Managed Services)
  • Processing customer payments across agreed payment channels
  • Credit collection activities
  • Wholesale energy trading and market reconciliations
  • Network access arrangements and network reconciliations 
  • Compliance monitoring and reporting to the AER


    • In Scope

  

CS Energy is an existing client who has been utilising Tally CIS since 2021.  CS Energy utilise Salesforce for a number of functions and these integrate with Tally CIS.  CS Energy will be replacing Salesforce and the scope of this Blueprint is to define the functions that will move into Tally CIS.  These functions will include:

  • Some Sales Management functions
  • Case Management including general enquiries, onboarding, complaints and EWR.
  • Some Credit Control functions including dunning notice generation.

Availability:

Tally provides high availability levels:

As a Microsoft Azure hosted solution Tally provide an upper threshold of availability of data which does not exceed the AZURE SLA of 99.9% during operational hours (6am-6pm AEST).  

AZURE has near seamless failover with worst case scenario of users being logged out and required to login again.

The Tier1 level of redundancy and failover well mitigates the following: 

power outage, single points of failure, unavailability of critical staff or unsatisfactory maintenance of equipment, and failure of equipment/software.

NF-026

Security

Data in Motion:

Azure SQL Database always enforces encryption for all connections, which ensures all data is encrypted when transiting between the database and database clients. Data stored in Azure Blob store also enforces TLS 1.2 for all data transfers to and from the blob store. Between CS Energy and API/application layer all traffic is encrypted using TLS 1.2, this also includes traffic from CS Energy to the Web Application Firewall (WAF) and between the WAF and API/application layers. As a guiding principle all traffic between and within the Tally stack is encrypted.








NF-027

Security

Data at Rest:

NF-028

Security

Threat Detection:

Threat Detection complements auditing, by providing an additional layer of security intelligence built into the Azure SQL Database service that detects unusual and potentially harmful attempts to access or exploit databases. Alerts about suspicious activities, potential vulnerabilities, and SQL injection attacks, as well as anomalous database access patterns are provided. Threat Detection alerts can be viewed from Azure Security Centre and provide details of suspicious activity and recommend action on how to investigate and mitigate the threat. Tally also uses AZURE Front Door to act as a web application firewall to mitigate against DDoS attacks and other potential vulnerabilities. All access is logged on the Azure platform. Tests are conducting against the OWASP Top 10 and CWE Top 25 frameworks. Tally can share reports with clients as part of our account management process.


Azure also scans and classifies data it considers sensitive –e.g., address information, phone numbers, email address. A vulnerability assessment of data access is generated and reported on.

NF-039

Data Management

Tally CIS uses a ‘Database’ per tenant model.  In this model the data for each tenant is separated into its own database container. This ensures that 

  • Data is not accidentally ‘leaked’ due to permission errors.  In a Single database tenant model, all the data must be tagged with the data’s owner. Security is dependent on API’s correctly filtering all data requests with this ownership key. With a multi-DB model, the API connection determines the owner. Generally, the connection is done once and in one spot- whereas a large app will have hundreds of data access points – all of which must correctly filter.
  • In Azure each database can share or have its own scalability settings – this allows us to isolate a DB and avoid the ‘noisy neighbour’ problem
  • Data replication can be used to create separate read-only DBs for things like reporting.

While the DB is separate to each tenant, the DB structure (schema) must be the same as the code is identical for all clients. Tally uses our CI/CD pipeline to ensure DB changes are applied to all DB’s.








NF-040

Data Management

Environments:

Given the code base is the same, Tally uses ‘Data driven’ techniques to provide customization.  These are some of the techniques that Tally uses;

  • Options – we use options to configure various logic in the system. Most options can be set at various levels from the account to the overall top-level tenant. An example of an option would be the name of the Invoice Template to use, or the max threshold to hold a bill. An option will inherit from its parent such that for an account X the system will 
    • First look at any specific options for X 
    • If none then look at its parent account 
    • If none then look at the business unit 
    • If none then look at the tenant option.  

This means that options set for the tenant will be applied to all accounts unless they have been overridden at a lower level.

  • Feature Flags – Feature flags are used to turn parts of the system on or off- For example we could use a feature flag to turn quoting on or off. If off the UI for the CIS would not show the feature at all – e.g., it would not be visible. 
  • Product calculations - Each account may have its own set of products and calculations – this allows for the creation of bespoke calculations. For example, for ‘unaccounted for energy’ (UFE) calculation and product, a tenant would not have to apply this product – this by configuring the products on a customer – the customer controls the invoice creation.  
  • Templates: Invoices and other documents use “templates”. A template is like a mail merge document – a common data payload is combined with the template.
  • To produce items like the invoice PDF, templates can have complicated commands and logic in them which provides a large amount of flexibility.

The standard offering provides for Point in time Recovery. Account Creation

Once a contract is accepted, CS Energy will manage the account creation via the Onboarding UI (Add New Accounts) for bulk uploads or alternatively the Manual Creation Wizard can be utilised for once off account creation.

The Add New Accounts wizard enables a user to efficiently create a single file upload which may contain multiple accounts. While attaching contacts during the process, users can associate contacts with accounts by inheriting contacts from an existing account, searching for existing contacts, or creating new contacts where required. The wizard supports assigning more than one contact type; however, the same contact role configuration is applied consistently across all accounts in the uploaded file. Where different contact roles are required for individual accounts, these must be updated manually after account creation. The File Specification for the Add New Accounts template has been supplied to CSE.

The product association is handled in a similar manner. Products selected during the wizard are applied consistently across all accounts in the file. Any account-specific product differences must be managed post-creation.   With regards to new rate cards these can be created prior to the Add New Accounts wizard and then select the ratecard during the wizard, alternatively the user can skip the product assignment and create & assign  the new rate card post account creation.

Enhancement

The Add New Accounts UI file specification must-

  • support the ability to specify whether Non-Financial EDI (NFEDI) is required at the time of account creation
  • enable ABN/CAN as a mandatory field for account creation 

The wizard will allow users to create parent accounts that do not have an associated NMI and to create child accounts linked to a parent account. This capability is supported through both the Add New Accounts UI and the manual account creation wizard. 

Enhancement

Introduce validation controls to ensure that child account must have a Primary Contact and a CDN Contact recoded before account creation can be completed.

Introduce validation controls to ensure that where LS is flagged for the account/site, an LS Contact must be recorded before account creation can be completed

Tally CIS enforces different validation rules for Life Support and CDN contact types to ensure compliance and continuity of critical contacts.

For CDN contacts, the system ensures that a CDN contact is always present. Where a CDN contact is removed, a warning is displayed prompting the user to create or edit a CDN contact, or retain the Primary Contact as the CDN contact.

For LS contacts, stricter validation applies. An LS contact cannot be deleted directly. Where removal is required, the contact type must first be edited and changed to a non-LS role (for example, Secondary or Additional Contact). Only once the LS designation is removed does the system allow deletion. 

These tools will be used for new accounts, greenfield sites and contract roll ins.  CS Energy will use the NMI Discovery default for the population of the site address.

New connection workflow is initiated when a new account is created and flagged as a new connection. When this happens, a New Connection tag displays in the Account > Overview page.  


The new connections workflow is largely automated. It combines a sequence of service orders and market transactions which transform an unserviced site to a metered, supplied and importantly, billable account. The process is defined in a sequence of steps which are further organised in a series of stages.


Stage

Steps

1. Prepare
  • Wait for Release from Hold
  • Validate Data
  • Welcome Pack is not required for CS Energy

2.  New
  • Wait for Electrical Work Request (EWR)
  • Send Allocate NMI Service Order Request

3.  Allocate NMI in progress
  • Wait for Allocate NMI response or NMI with Standing Data Masters exists on account
  • Re-Raise Allocate NMI Service Order Request – if original SO is not completed or rejected A Not Completed response is considered very unlikely
  • Wait for re-raised Allocate NMI response or NMI with Standing Data Masters

4.  Allocate NMI Late
  • If No SO Response received within 5 Business days

5. Allocate NMI Complete
  • Send Change Customer Classification Code (CCC) Change Request
  • MRF is required to be sent for CS Energy
  • Send Establish Supply Service Order Request

6. Install in Progress
  • Wait for Establish Supply response or NMI status to A and account to have actual start date set
  • Re-raise Establish Supply Service Order Request - if original SO is not completed or rejected
  • Wait for Re-raised Establish Supply Service Order response or NMI status to A and account to have actual start date set

7. Install Late
  • NMI status is not Active within set Threshold Days (120 days)

8. Install Complete
  • NMI status is active and actual start date set for the account within 120 days

9. Finalisation
  • Send Life Support Notification
  • Send Customer Details Notification

10. Completed
  • Life Support Notification sent
  • Customer Details Notification sent

PC-006

Pricing

Tally CIS supports Spot Pricing Products:

  • For E channel consumption, where that contract has a spot pricing retail product.
  • For all contracts that have B channel consumption.  That may mean the E channel could have a TOU Retail tariff and the B channel will have a spot pricing product.  Alternatively, both the E and B channel can have the same spot pricing product type.


The spot pricing products may be 

  • Spot price pass through with no administration fee.
  • Spot price pass through, plus an administration fee in $/MWh, where these are calculated and displayed on the bill as two separate line items in the Energy Charges section.
  • Spot price pass through, plus a percentage administration fee.  This is calculated and displayed on the bill as a bundled line item in the Energy Charges section.
  • Where the site has both E channel consumption and B channel generation, where both channels usage is being charged at pool price there will be separate lines for export and consumption with the bill display to show the nett result.  Only one administration fee will be charged. This will be charged as a separate line item in $/MWh. 
  • Spot pricing will have MLF and DLF applied.  

Spot pricing is based on the RRP in the jurisdiction where the supply address is located.








PC-007

PC-008

Pricing

The OOTB UFE functionality consists of:

  • UFE is only applicable to distribution connected customers.  
  • UFE is calculated monthly based on a load weighted average per week based on RM data from AEMO. An estimate is created for the remaining days of the month where AEMO settlement data is not available. The average is calculated for Peak and Off-Peak periods. 
  • Washups to UFE calculations are performed as settlement data revisions are issued, wash ups are approved by users for application to the bills.
  • The UFE bill line item will appear in the Energy Charges section of the bill.  The UFE will add to the sites overall energy consumption and therefore be the basis of any other energy based charges such as AEMO Pool Fees, AEMO Ancillary Service Charges, Renewable Certificates.








PC-009

Pricing

  • Swap (MW to be configured).

  • Spot price.


The bill design will display in the Energy Charges section, Spot Pricing Anytime, Swap value and a credit for the cap value (where cap price reached).

BL-018

Bill Calculation

Billing of an on market embedded network parent site:

  • The retail component is calculated as the net (with losses) of the parent minus the embedded children.
  • UFE is also calculated. 
  • Will include solar generation where applicable.
  • The network charges are calculated on just the parent usage -without the child sites subtracted. 
  • The market charges are based on the net 
  • The following calculation is used to determine the Parent Retail Charge

Parent(E) = per interval (MAX(Parent E*Losses – Child(n) e*Losses,0)) + (MAX(Child(n) b*Losses – Parent B*Losses,0))

Parent(B) = per interval (MAX(Parent B*Losses – Child(n) b*Losses,0)) + (MAX(Child(n) e*Losses – Parent E*Losses,0))

  • Losses are handled within the Site calculations as Embedded Networks can use multiple Loss Factors


The Tally CIS Bill Explorer displays: Sub-sites that have been subtracted from the parent, Meters used including the To and From dates, and meter data used and each child read subtracted from the parent.

BL-032

Bill Calculation

Bill reversals (voids) and rebills:

  • Cancelled bills must be reversed in sequence e.g., reversing a bill from 3 months ago will require all bills from this date onwards to be reversed and reissued. 
  • A cancelled invoice is in effect a credit note and will appear on the customer’s consolidated statement.  A cancelled invoice is linked to the original invoice.
  • The customer account balance is credited on cancellation.
  • When voiding an invoice, by default a reason is required to be entered.
  • Bill reversals can be limited to a specified number of months. By default, this is configured to 9 months.
  • To bill a fixed charge, post a final account the final invoice must be reversed and the charge added and rebilled.
  • A Tax Adjustment Note is currently not supported

PR-008

Manage Payments

Due to the type of CSE customer base, dishonoured Direct Debits do not occur.


Tally CIS does support a dishonour process where the bank dishonour code is processed indicating a permanent or temporary dishonour reason.   


  • A permanent dishonour is an unresolvable issue, such as an invalid account number. In these cases, the Direct Debit arrangement can be cancelled, and the Payment Method will be set to ‘External/Other’ in Tally CIS.     A Direct Debit Cancelled notice can be triggered in Tally CIS and emailed to the customer.  The document and customer interaction will be recorded on the customer’s account, together with updating the Payment Method to unknown.  Dunning will commence from the first step the next business day based on the invoice due date.
  • A temporary dishonour refers to a temporary issue with the payment such as insufficient funds.  A retry process is configurable, which enables the direct debit to be retried, and a  Dishonour Direct Debit notice sent to the customer.  A customer interaction will be recorded on the customer’s account for the dishonour.  
  • A list of dishonour codes is maintained in the Advanced organisational settings configuration.  This is viewable by authorised users.  
  • In addition, where the customer continues to have repeat dishonours across multiple bills, configuration can restrict the number of consecutive dishonours before the direct debit arrangement is cancelled.

  • Prevent disconnection. 
  • Prevent dunning. 
  • Allow fees and charges to be applied.

Tally CIS provides configuration to support payment extensions. This will be configured for CS Energy as:

  • Maximum days an invoice due date is allowed to be extended (15 calendar days)
  • Maximum times an invoice due date is allowed to be extended (3 in a year)
  • Number of months to consider for max number of due date extensions (12 months)
  • When a payment extension finishes, the original due date remains for the start of the dunning process.

  • Appendix K - Guideline for Tally’s Client to integrate with SendGrid


Important: If CS Energy is currently having SendGrid integration, Tally implementation team will need to be informed so that a custom DKIM will be generated.


1. Tally’s Client to add SPF Record 

To configure SPF for Tallys’s Client domain to authorise SendGrid to send emails on client behalf:

a) Domain’s DNS Management Settings:

  • Access the DNS provider’s management console (this could be with the web hosting provider, domain registrar, or a dedicated DNS management service).

b) Add/Update the SPF Record:

  • Locate the existing SPF record for the domain, if any. If none exists, client need to create a new TXT record.
  • Add or update the SPF record to include SendGrid's mail servers.
  • Example of the configuration of SPF record:

If no SPF record yet:

Add a new TXT record with the following value:
 v=spf1 include:sendgrid.net -all

If SPF record exists:

Modify the existing record to include SendGrid. For example:
 v=spf1 include:sendgrid.net include:otherdomain.com -all


Be cautious not to exceed the SPF 10 DNS lookup limit.


Save Changes:

Save the TXT record to update your DNS settings.


2. Tally’s Client to Configure DKIM and DMARC (optional)

a) DKIM (DomainKeys Identified Mail):

  • In SendGrid, under Domain Authentication, Tally team will provide the DKIM records to add to the DNS settings.
  • Add the provided DKIM TXT records to the domain’s DNS.


b) DMARC (Domain-based Message Authentication, Reporting & Conformance):

  • DMARC helps to specify how email from Tally client’s domain should be handled if it fails SPF or DKIM checks.
  • Add a DMARC TXT record to the DNS with a policy, e.g.:
         _dmarc.yourdomain.com. IN TXT "v=DMARC1; p=none; rua=mailto:dmarc-reports@yourdomain.com"
  • Adjust the policy (p=none, p=quarantine, or p=reject) based on how strict you want the handling of emails that fail DMARC checks.


3.  Testing

a) Send a Test Email:

  • Use SendGrid to send a test email and check the email headers to ensure SPF, DKIM, and DMARC are correctly applied.


b) Use Online Tools:

  • Utilize tools like MXToolbox, DMARC Analyzer, or other email testing tools to verify SPF, DKIM, and DMARC records are set up correctly.
