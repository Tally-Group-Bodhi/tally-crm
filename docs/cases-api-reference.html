1. API Endpoints

All case endpoints are served from the Next.js API under /api/cases.

GET /api/cases
List all cases. Optional query param ?caseNumber=X to filter by case number.
POST /api/cases
Create a new case. Body: full CaseItem JSON (see fields below). Returns the created case.
GET /api/cases/[id]
Fetch a single case by its id (CUID).
PATCH /api/cases/[id]
Update a case. Body: partial CaseItem — only include the fields you want to change. Returns the full updated case.
DELETE /api/cases/[id]
Delete a case by id.
POST /api/cases/[id]/attachments
Upload file(s). Body: FormData with a files field. Returns Attachment[] metadata. Note: this does not update the case record — the client must then PATCH the case with the updated attachments array.
GET /api/cases/[id]/attachments/[attachmentId]
Download or view an attachment. Query param ?inline=true for inline display (e.g. PDF preview).

4. Status & Progress Workflow

The UI progress bar shows this linear flow:

New
→
In Progress
→
Pending
→
Closed
No enforced state machine: The API does not enforce status transitions. Any status can be set to any other status via PATCH. The UI guides users through the flow above, but the API accepts any valid CaseStatus value.

Transition Details

TransitionHow It HappensExtra Data
New → In ProgressStatus bar click, Kanban drag, or PATCHNone
Any → PendingStatus bar click, Kanban drag, or PATCHpendingReason: "Customer" | "3rd Party" | "On Hold"
Any → ClosedClose Case modal or PATCHresolution string + close reason
Any → ResolvedPATCHresolution string

Updating Status via API

// Minimal PATCH to move a case to "In Progress"
PATCH /api/cases/{id}
Content-Type: application/json

{
  "status": "In Progress"
}

// Move to "Pending" with reason
{
  "status": "Pending",
  "pendingReason": "Customer"
}

// Close a case
{
  "status": "Closed",
  "resolution": "Issue resolved after billing adjustment applied."
}

5. Case Number Format

Case numbers follow the pattern: PREFIX-YYYY-NNNNNN

PartDescriptionExample
PREFIXDerived from account name: first letter of each word (up to 4 chars), or first 4 chars for single-word names. Uppercase."Big Box Mega Corp" → BBMC
YYYYFour-digit year2026
NNNNNNSix-digit zero-padded sequence number001847

Full example: BBMC-2026-001847

Important: Case numbers must be unique. If integrating from an external system, coordinate numbering with the CRM to avoid collisions. The generation logic lives in lib/case-number.ts — function generateCaseNumber(accountName, sequence, year?).

11. Current Gaps & Integration Notes

No Account/Contact/Org API: These entities have no database tables or CRUD endpoints. To push this data from an external system, you would need to either:
  1. Add Prisma models and API routes for Account, Contact, and Organization
  2. Or continue denormalising — just pass accountId and accountName when creating cases
Communications & Activities are append-only arrays: To add a new note/email/activity, you must GET the case, append to the array, then PATCH the entire array back. There is no dedicated "add communication" endpoint.
Attachment upload is two-step: First POST the file to /api/cases/[id]/attachments, then PATCH the case with the returned attachment metadata merged into the attachments array.
No authentication: The API currently has no auth layer. All endpoints are open. Factor this in when planning external integration.
Database fallback: If DATABASE_URL is not set, the API returns 503 and the app falls back to in-memory mock data. Ensure the database is configured for production use.
