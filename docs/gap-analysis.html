Executive Summary

The CSE Blueprint describes a full enterprise CIS (Customer Information System) platform for an electricity C&I retailer. The current Tally CRM is a UI prototype focused on case management and pipeline visualisation with mostly mock data and a single database table (Case). The vast majority of required functionality — particularly energy industry-specific features — has not been built.

1. Sales Management

~90% missing
What exists: A basic pipeline/opportunity UI with Kanban and list views, KPI cards (win rate, pipeline value), and a new opportunity modal. All backed by mock data with no database persistence.

Missing Functionality

  • Opportunity data model & persistence — no database table. Blueprint requires detailed fields: Broker Name/Contact, Customer ABN, ASIC Industry code, NMI, annual load forecast, generation capacity, current retailer, tender due date, contract dates, etc. SM-001
  • NMI Discovery integration — automatic AEMO NMI lookup for all NMIs within an opportunity, with exception logging for mismatches. SM-002
  • ABN lookup integration — automated ABN validation against the Australian Business Register on opportunity creation. SM-003
  • Opportunity cancellation — cancellation date and reason recording. SM-004
  • Quote lifecycle workflow — entire multi-step quoting lifecycle: product type selection (9 energy product types), credit check triggers to Finance Team, pricing requests to Pricing Team, pricing loading (UI or file upload), quote status management (New → Pricing Requested → Pricing Completed → Negotiation → Contracting → Closed Won/Lost). SM-005 to SM-016
  • Quote & Pricing Schedule correspondence — auto-generated quote documents with mail merge fields (customer name, NMIs, contract dates, pricing summary). SM-019
  • Contract generation — automatic contract production from accepted quotes, Docusign/e-sign support, contract adjustment before sending. SM-015, SM-020
  • Closed Lost tracking — configurable close reasons dropdown, successful retailer selection, accepted term recording, close notes, automatic future opportunity creation 6 months prior to contract expiry. SM-016
  • Pricing withdrawal & reprice — Pricing Team can withdraw pricing at any point; Sales Manager can trigger repricing on original quote. SM-013, SM-014
  • Sales reporting — Number of RFPs, contract lost volumes and reasons. SM-021
  • Customer Onboarding API — integration point for account creation from won opportunities. SM-022

2. Customer Management

~85% missing
What exists: Basic Organization, Account, and Contact entities with table and card views. Case management with CRUD, multiple views (list, kanban, tab), filtering, SLA indicators, case linking, file attachments, and activity/communication timelines. All customer data is mock; only the Case model has a database table.

Account Creation & Onboarding

  • Full account data model — no persistent Account table. Blueprint requires ABN/ACN, account types (C&I, Occupier), external references, parent/child/grandparent hierarchies, NMI associations, contract dates, site attributes, billing addressee, consolidation flags. CS-001 to CS-025
  • Add New Accounts bulk upload wizard — file upload for multi-account creation with contact/product inheritance, hierarchy view, validation controls for contacts (CDN, LS, Primary). CS-001
  • Account numbering — configurable auto-numbering series with format rules. CS-003
  • Market transfer automation — auto-trigger CR1000 and NMIStandingDataRequest to MSATS upon account creation (up to 65 business days in advance). CS-008
  • Parent/child hierarchy management — contact inheritance, flag inheritance, terms inheritance, ability to move child to different parent. CS-016, CS-017
  • Occupier account management — auto-creation of occupier accounts when CS Energy remains FRMP after account closure. CS-024
  • Direct Debit management — flags, confirmation notices, bill template triggers, payment terms configuration. CS-013
  • Welcome Pack correspondence — auto-generated introduction letters with portal info, account terms, NMI, parent account number. CS-022
  • Account closure workflow — Close Account wizard with de-energisation options, service order types, final billing, occupier auto-creation. CS-010

Life Support Module

None of this exists. CS-029 to CS-040
  • Life support correspondence — information pack, medical confirmation, initial/second reminders, extension confirmation, deregistration notices (7 separate templates). CS-083 to CS-090

Sensitive Load

  • Sensitive Load flagging — checkbox, icon display in account header, CDN/CSDN auto-triggers to distributors, credit control tag management to prevent disconnection and dunning. CS-027, CS-028

Case Management Enhancements

  • Email inbox integration — auto-create cases from monitored CS Energy email inbox with parsing logic, field-mapping, and validation rules. CS-041
  • Web form integration — create cases from structured form submissions (EWR, enquiry forms, B2B Portal). CS-041
  • Case Class — mandatory classification (Complaint, General Enquiry, etc.) driving downstream system behaviour. CS-042
  • Auto-acknowledgement emails — automatic email on case creation with case number, inside and outside business hours variants. CS-043
  • Queue/group assignment — cases assigned to operator groups/queues with configurable rules; reassignment warnings when attempting to action another user's case. CS-044
  • Configurable contact reasons — mandatory primary contact reason per case type for reporting and SLA measurement. CS-045
  • Recurring & future-dated cases — scheduled case creation and future-dated cases with defined start dates. CS-046
  • Portfolio/hierarchy linkage — parent-level visibility of all portfolio cases; child-level limited to site; Customer Type field. CS-047
  • Service Order integration within cases — network notices (Planned Interruption), EWR workflows, dunning flows managed as case types. CS-048
  • Action panel in cases — performing Tally CIS action wizards directly from case context (update contacts, upload documents, bill enquiries). CS-049
  • Bidirectional email in cases — incoming email parsing and linking to existing cases, configurable templates with mail merge, resend capability, email detail visibility. CS-050
  • Parent-child case structuring — grouping related issues under a broader case, manual or rule-based grouping. CS-051
  • Document management enhancements — document expiry dates, central Documents view, navigation from case to Documents tab. CS-054
  • DOA and escalation matrices — Delegation of Authority and escalation support for defined case management stages. CS-057
  • Contact search in case creation — search by contact showing existing cases and contact type/role. CS-058

Contract Management

  • Contract lifecycle — renewal monitoring via opportunity management, re-contracting (new contract creation, product/rate card assignment), default rate fallback with exception raising, configurable payment terms (14, 21, 30, 45 calendar days). CS-059 to CS-061

Customer Enquiry & Portal

  • Bill dispute management — bill holds with dispute reason, credit control tag of Dispute. CS-066

  • Bulk interaction file upload — add interactions for one or many accounts via spreadsheet. CS-077

  • Bulk email capability — send to customer groups or distribution lists with corporate signatures and self-service templates. CS-074

8. Non-Functional & Infrastructure

Significant gaps
  • Authentication & SSO — no OpenID Connect, no federated identity with Azure AD, no Auth0 integration. Portal and CIS both require SSO. NF-033, NF-034
  • RBAC enforcement — settings UI exists but permissions are not enforced on API routes or page access. NF-030
  • Audit logging — no real audit trail for record changes; blueprint requires detailed change tracking with 12-month retention across 20+ tables. NF-036, NF-037
  • Data export — no CSV/Excel export capability from data tables. GA-003
  • SendGrid integration — no email sending capability for correspondence, billing, or ad-hoc communications. GA-012, NF-024
  • Webhook system — no webhook events for external integration (market transactions, account events, invoice status, payments, etc.). NF-019
  • Comprehensive API suite — only basic case CRUD APIs exist. Blueprint requires public APIs for onboarding, market data, meter data, rate cards, portal, billing adjustments, and more. NF-017
  • Database schema — only a single Case model in SQLite. A full relational schema is needed for accounts, contacts, contracts, sites, meters, products, rates, invoices, payments, interactions, activities, life support records, service orders, and more. Multiple
  • Multi-environment support — no UAT/Production separation, no environment-specific email safeguards. NF-010, GA-008
  • Azure cloud hosting — currently a local Next.js app with SQLite; blueprint expects Azure SQL, Azure Blob Storage, Azure Front Door WAF, geo-replication. NF-005 to NF-009, NF-047

Note: This analysis compares the current Tally CRM codebase against the CSE Blueprint Replacing Salesforce v0.1 document dated 16 January 2026 (with updates through 11 February 2026).
